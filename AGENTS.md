# AGENTS.md

## 项目概述

| 项目    | 说明                                                                                   |
| ----- | ------------------------------------------------------------------------------------ |
| 仓库名称  | `skills-repository`                                                                  |
| 产品名称  | `skills管理器`                                                                          |
| 产品定位  | 跨 Agent 的 skills 资产管理层 / 控制平面                                                        |
| 当前阶段  | 文档驱动阶段，当前仓库以方案、原型、调研、任务拆解为主，不是完整应用代码仓库                                               |
| 目标用户  | 多 Agent 个人开发者，Windows 优先                                                             |
| 目标形态  | 桌面管理器，未来实现阶段默认采用 `React + TypeScript + Vite + Rust + Tauri`                          |
| UI 栈  | `Tailwind CSS + DaisyUI + HackerNoon Pixel Icon Library`                             |
| 本地存储  | `文件系统 + SQLite`                                                                      |
| 当前主产物 | `PRD.md`、`TechDesign.md`、`PrototypeDesign.md`、`PrototypePromptPack.md`、`Compet-PRO/` |
| 协作原则  | 先统一文档认知，再做计划，再进入原型或代码实现                                                              |

## 响应与协作约定

| 规则       | 要求                                       |
| -------- | ---------------------------------------- |
| 默认语言     | 默认使用中文                                   |
| 输出形式     | 结论先行；计划、总结、规范类内容优先用表格，便于 agent 扫描        |
| 事实确认     | 不把猜测写成事实；涉及外部最新技术、官方能力、版本变更时优先查官方资料      |
| 重要结论沉淀   | 会影响产品、设计、技术或实现边界的结论，必须回写对应真源文档，不能只停留在对话里 |
| 优先修改已有文件 | 若已有承载位置，优先更新原文档，避免创建重复语义文件               |

## 仓库现状与边界

| 事实           | 说明                                                                 |
| ------------ | ------------------------------------------------------------------ |
| 当前不是完整业务代码仓库 | 不要擅自假设已经存在 `src/`、`src-tauri/`、数据库迁移、Tauri commands 或可运行应用         |
| 文档是当前真源      | 当前任务大多应落在需求、技术设计、原型、任务拆解与竞品研究层                                     |
| 视觉参考不等于正式实现  | `桌面应用设计风格定位.tsx` 仅是视觉探索参考，不应被当作正式业务代码真源                            |
| 竞品资料不是项目事实   | `Compet-PRO/` 只能提供启发，不能替代项目结论                                      |
| 未来实现约束不等于已实现 | `React + Rust + Tauri`、`文件系统 + SQLite`、i18n、主题等是已确认方向，但不是当前仓库已存在能力 |

## Source of Truth

### 优先级顺序

| 优先级 | 文件                       | 用途                        |
| --- | ------------------------ | ------------------------- |
| 1   | `PRD.md`                 | 产品目标、范围边界、功能定义、用户场景、优先级   |
| 2   | `TechDesign.md`          | 技术选型、模块边界、数据模型、接口、流程、测试口径 |
| 3   | `PrototypeDesign.md`     | 页面结构、信息架构、组件层次、交互说明       |
| 4   | `PrototypePromptPack.md` | 面向 AI 设计工具的原型生成提示词        |
| 5   | `Compet-PRO/`            | 竞品调研与技术参考，不代表本项目已决策内容     |
| 6   | `桌面应用设计风格定位.tsx`         | 视觉探索参考，不是正式实现真源           |

### 任务对齐规则

| 任务类型       | 先读什么                                          | 默认落点               |
| ---------- | --------------------------------------------- | ------------------ |
| 产品需求新增/变更  | `PRD.md`                                      | `PRD.md`           |
| 技术方案新增/变更  | `TechDesign.md`                               | `TechDesign.md`    |
| 页面结构/交互/原型 | `PrototypeDesign.md`、`PrototypePromptPack.md` | 对应原型文档             |
| 竞品问题       | `Compet-PRO/`                                 | 调研文档或引用说明          |
| 研发任务拆解     | `PRD.md` + `TechDesign.md`                    | `TASKS.md` 或专项计划文档 |
| 正式开始编码     | `TechDesign.md` 必须已足够支撑实现                     | 代码目录或后续实现目录        |

### 冲突处理

| 场景          | 处理方式                                       |
| ----------- | ------------------------------------------ |
| 用户需求与现有文档冲突 | 先明确指出冲突，优先建议更新真源文档，再继续扩展或实现                |
| 文档之间不一致     | 先修正 `PRD.md` / `TechDesign.md` 的冲突，再开展下游工作 |
| 文档缺口影响实现    | 不要静默补需求；先补文档，再编码                           |

## 工作流程

| 阶段      | 要求                                     |
| ------- | -------------------------------------- |
| 1. 读取真源 | 先读相关 `PRD.md` / `TechDesign.md` / 原型文档 |
| 2. 判断缺口 | 判断是否存在需求缺口、技术边界缺口或文档冲突                 |
| 3. 回写真源 | 任何影响产品或技术结论的信息都应先写回文档                  |
| 4. 输出计划 | 在文档对齐后，再输出任务拆解、原型说明或实施计划               |
| 5. 进入实现 | 只有在文档已可执行时，才进入代码或高保真原型落地               |

## 开发规范

### 通用工程规范

| 规则     | 要求                                         |
| ------ | ------------------------------------------ |
| 文档优先   | 先对齐文档，再进入实现                                |
| 显式决策   | 不要静默补需求、补流程、补边界                            |
| 不做静默降级 | 不新增 silent fallback、mock success、隐式兜底来掩盖问题 |
| 失败要可见  | 优先显式报错、异常、日志和失败测试，便于定位根因                   |
| 保持简洁   | 遵循 SOLID、DRY、分层清晰、YAGNI                    |
| 删除陈旧路径 | 当行为已经明确变化时，及时清理废弃逻辑和过期兼容代码，除非文档明确要求保留      |

### 架构边界

| 层级              | 应负责                     | 不应负责                     |
| --------------- | ----------------------- | ------------------------ |
| React 页面层       | 布局、交互、状态呈现、表单校验、路由切换    | 不直接访问本地文件系统，不直接访问 SQLite |
| Zustand 层       | UI 状态、筛选条件、任务面板、设置镜像    | 不承载复杂业务规则                |
| Tauri command 层 | 参数解析、权限判断、任务创建、响应封装     | 不写具体业务细节                 |
| Service 层       | 扫描、安装、分发、模板注入、安全扫描、更新治理 | 不写 SQL 拼接，不关心 UI 展示      |
| Repository 层    | SQLite 事务、表读写、缓存管理、日志写入 | 不接触外部 API 和文件系统业务编排      |
| Adapter 层       | 市场源归一化、Agent 路径能力矩阵     | 不操作数据库                   |

### 前端规范

| 规则         | 要求                                                    |
| ---------- | ----------------------------------------------------- |
| Tauri 调用入口 | 所有调用统一通过 `src/lib/tauri-client.ts` 封装                 |
| 页面组件限制     | 不允许在页面组件中直接 `invoke()`                                |
| 任务监听顺序     | 任务型命令必须先注册事件监听，再触发 command                            |
| i18n       | 用户可见文案统一走 i18n key，不写死字符串                             |
| 语言范围       | 首版默认支持 `zh-CN`、`en-US`、`ja-JP`                        |
| key 命名     | 使用 `页面.模块.语义` 形式，例如 `settings.theme.mode.light`       |
| 主题         | 统一使用 DaisyUI token 与 `html[data-theme]`，禁止写死颜色值       |
| 图标         | 使用 HackerNoon Pixel Icon Library 时默认继承 `currentColor` |

### Rust / Tauri 后端规范

| 规则         | 要求                                                 |
| ---------- | -------------------------------------------------- |
| command 设计 | `#[tauri::command]` 仅做参数解析、权限判断、任务创建和响应封装          |
| 业务分层       | 扫描、安装、分发、模板、更新、安全逻辑放在 `services/*`                 |
| 数据访问       | SQLite 仅通过 repository 层访问；前端不得直连数据库                |
| Agent 路径真源 | `AgentRegistry` 仅在 Rust 维护，前端通过 `bootstrap_app` 获取 |
| 分发策略       | 全局分发优先 `symlink`，项目注入优先 `copy`                     |
| 权限失败处理     | Windows 下 `symlink` 失败时必须显式报错，不自动静默降级为 `copy`      |
| 路径安全       | 删除、覆盖、分发前都要 `canonicalize`，只允许在白名单根目录内操作           |
| 错误处理       | 使用可解释错误码与明确失败原因，不吞错误                               |

### 数据与安全规范

| 规则              | 要求                                        |
| --------------- | ----------------------------------------- |
| canonical store | 所有受管 skill 优先进入统一仓库，再进行分发                 |
| 本地存储            | skill 实体放文件系统；元数据、模板、设置、日志、安全报告放 SQLite   |
| 安全扫描            | `SKILL.md`、`scripts/`、常见脚本、可执行文件纳入扫描范围    |
| 阻断规则            | `high/critical` 风险默认阻断，不落 canonical store |
| 日志与诊断           | 下载中断、模板失败、路径越界、权限失败等场景要留下可诊断信息            |

## 代码风格

### 通用风格

| 类别   | 要求                                |
| ---- | --------------------------------- |
| 命名   | 清晰、语义化，避免缩写和模糊命名                  |
| 注释   | 只为关键或不直观逻辑补充简洁说明，不写噪声注释           |
| 复杂度  | 关注路径扫描、文件复制、模板注入等 IO 密集逻辑的时间与空间成本 |
| 失败处理 | 显式处理边界情况，不隐藏失败                    |
| 废弃代码 | 修改行为时同步移除无用逻辑、过时字段、失效兼容分支         |
| 单文件  | 单一职责原则、高内聚、低耦合、不要臃肿               |

### 文档风格

| 规则   | 要求                                                                                            |
| ---- | --------------------------------------------------------------------------------------------- |
| 语言   | 默认中文                                                                                          |
| 结构   | 长内容优先表格，结论先行，便于 agent 快速扫描                                                                    |
| 文件命名 | 文件名直白、语义明确；优先放根目录，除非天然属于子目录                                                                   |
| 现状描述 | 不能把“计划中”“建议中”“未来约束”写成“当前已实现”                                                                  |
| 联动更新 | 更新 `PRD.md` 后，如影响技术方案或原型，必须同步检查 `TechDesign.md`、`PrototypeDesign.md`、`PrototypePromptPack.md` |

## 测试要求

### 文档任务验收

| 类型     | 最低要求                                                         |
| ------ | ------------------------------------------------------------ |
| 需求文档修改 | 检查与 `PRD.md` 的目标、范围、非目标一致                                    |
| 技术方案修改 | 检查与 `TechDesign.md` 的架构、接口、存储、测试口径一致                         |
| 原型文档修改 | 检查与 `PrototypeDesign.md`、`PrototypePromptPack.md` 的页面结构与交互一致 |
| 结论回写   | 重要决策不得只存在于对话中，必须落文件                                          |

### 代码任务测试分层

| 层级        | 工具/方式                            | 必要说明                                                 |
| --------- | -------------------------------- | ---------------------------------------------------- |
| Rust 单元测试 | `cargo test`                     | 覆盖扫描归一化、模板规划、安全评分、market adapter                     |
| 前端单元测试    | `Vitest + React Testing Library` | 覆盖 Store、i18n hook、主题 hook、关键组件                      |
| 集成测试      | Rust `tempdir` + SQLite 临时库      | 验证 command -> service -> repository -> filesystem 闭环 |
| E2E       | `tauri-driver + WebDriver`       | 验证主流程打通、Windows 权限、多语言、主题切换                          |

### 必测场景

| 类别   | 必测内容                                      |
| ---- | ----------------------------------------- |
| 启动   | 首次启动、升级后启动、数据库迁移失败                        |
| 扫描   | 系统级扫描、项目级扫描、自定义根路径扫描、重复识别                 |
| 安装   | 市场安装成功、市场安装阻断、高风险失败回滚                     |
| 分发   | 全局 `symlink` 成功、Windows 权限失败、项目 `copy` 成功 |
| 模板   | 模板注入成功、模板部分失败、模板项缺失                       |
| 更新   | 有更新、无更新、更新后重新分发                           |
| i18n | 中英日切换、重启保持、缺失 key fallback                |
| 主题   | 跟随系统、手动切换、重启保持、暗色可读性                      |
| 安全   | 高风险阻断、`medium` 风险提示、重新扫描                  |
| 删除   | 删除分发、删除 canonical store、非法路径拒绝            |

### 完成前检查

| 规则        | 要求                         |
| --------- | -------------------------- |
| 不做无证据完成声明 | 没有实际检查结果，不要声称“已完成”“已通过”    |
| 自动化检查     | 只要仓库已具备测试或校验脚本，就应实际运行并记录结果 |
| 文档一致性     | 需求、技术、原型三层如果被影响，必须做联动核对    |

## 注意事项

| 类别         | 注意点                                                  |
| ---------- | ---------------------------------------------------- |
| 产品边界       | MVP 核心是本地资产治理，不是单纯做 skills marketplace               |
| 非目标        | 不要擅自扩展团队私有市场、组织权限、云同步、账号体系、复杂推荐                      |
| Windows 特性 | `symlink` 权限与 junction 行为存在差异，必须显式提示失败原因             |
| 市场适配       | provider 状态需隔离上报，不要让单一市场源故障污染全部搜索结果                  |
| 图标许可       | Pixel Icon Library 当前公开使用受 `CC BY 4.0` 约束，产品内需保留署名策略 |
| 国际化        | 日文与英文长度差异较大，布局和按钮宽度必须预留弹性                            |
| 主题         | 自定义组件禁止写死颜色；亮/暗主题都要可读                                |
| 真源一致性      | 重要新增结论必须回写文档，不能只在临时文件或聊天里保留                          |

## 交付标准

| 检查项  | 标准                                  |
| ---- | ----------------------------------- |
| 项目化  | 内容必须服务 `skills管理器`，不能写成泛化模板         |
| 一致性  | 不与 `PRD.md`、`TechDesign.md` 已确认结论冲突 |
| 可执行  | 下一位 agent 能据此判断先读什么、先改什么、什么时候进入实现   |
| 现状真实 | 明确当前是文档仓库，不伪造“已实现”能力                |
| 扩展友好 | 能覆盖未来产品、设计、技术、实现协作场景                |
